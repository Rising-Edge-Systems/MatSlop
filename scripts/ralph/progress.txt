## Codebase Patterns
- Layout uses Allotment panes; panel sizes persisted via window.matslop.layoutSet
- Panels use data-testid attributes for Playwright selectors
- Run `npm run build` to verify Vite/electron-vite production build
- Pre-existing typecheck errors in src/main/octaveProcess.ts (env.PATH) are baseline noise
- Integration/unit tests that need bundled Octave fail in dev envs without resources/octave
- Prefer extracting pure helpers into src/renderer/editor/editorTypes.ts and adding vitest unit tests in tests/unit/
- For HTML5 drag/drop E2E: dispatch DragEvents manually via page.evaluate with a shared DataTransfer — Playwright's native drag is flaky
- LiveScript cell data lives in LiveScriptEditor.tsx; serialized via parseLiveScript/serializeLiveScript in editorTypes.ts
- `.ls-cells-grid` is a 2-column CSS grid; each code cell emits paired sibling grid items (code-side at col 1, output-side at col 2). Full-width rows (drop zones, add-cell rows, markdown cells) use `grid-column: 1 / -1`. Hover highlight is driven by a shared `hoveredCellId` React state that applies `.ls-cell-hover` to both halves.
- Per-cell controls (gutter = drag handle + run button/spinner; actions = delete button) live inside `.ls-cell-code-side` (code column), not on the output side. `.ls-cell-actions` uses `opacity: 0` + hover to fade in, so Playwright clicks need `{ force: true }`.
- Live-script vertical scrolling is container-level on `.ls-editor` (overflow-y: auto). Per-cell `.ls-cell-output` uses `max-height: none; overflow-y: visible; overflow-x: auto` so tall outputs expand their grid row (the container scrolls) and wide text scrolls inside the cell. `.ls-cell-output-side` uses `overflow-x: hidden` to prevent wide content from widening its `minmax(0,1fr)` grid track and breaking row alignment.
- Bundled Octave helper scripts live in `resources/octave-scripts/*.m`. When wiring them to the running Octave process, add the directory to `extraResources` in package.json AND addpath it at Octave startup (or copy into the bundled site m/ path).
- Plot JSON schema lives in `src/main/plotSchema.ts`. Use `parsePlotFigure(json)` to validate; unknown series types degrade to `{type:'unknown', octaveType}` so UI can show fallbacks. Emit JSON in Octave with `matslop_export_fig(h)` — it ships its own JSON encoder (don't assume `jsonencode` exists).
- `plotly.js-dist-min` is a runtime dep but has no .d.ts — a minimal `declare module 'plotly.js-dist-min'` in `src/renderer/env.d.ts` covers `newPlot`, `react`, `purge`, `downloadImage`, `Plots.resize`. Use the pure `figureToPlotly(figure)` adapter in `src/renderer/editor/plotlyAdapter.ts` to produce `{data, layout, config}` — it never imports the Plotly runtime, so it's unit-testable in node. Mount via the `PlotRenderer` component, which lazy-imports Plotly via dynamic `import('plotly.js-dist-min')` to keep the 4.7 MB bundle out of the initial renderer chunk.
- Live-script cell figure capture (see `runCellStatements` in `LiveScriptEditor.tsx`) emits TWO markers per plot group: `__MSLP_FIG__:<line>:<pngPath>` (always) and `__MSLP_FIG_JSON_FILE__:<line>:<jsonPath>` (only if `matslop_export_fig` is loaded). Pair them by iteration order per anchor line. The JSON is written via `fopen/fwrite` in Octave (don't try to embed JSON directly in a disp() marker — it has newlines) and read by the renderer via the preload IPC `window.matslop.figuresReadTextFile(path)`.
- To make bundled Octave helper scripts available to the running process: pass `getMatslopScriptsDir()` (octaveConfig.ts) as the second arg to `new OctaveProcessManager(binaryPath, scriptsDir)`; the process's init script addpath()s it before returning from start(). The helper resolves `resources/octave-scripts` in both dev (`app.getAppPath()`) and packaged (`process.resourcesPath`) layouts.
- `LiveScriptCellFigure` carries an optional `plotJson?: string` field. `InlinePlots` prefers `PlotRenderer` (data-testid=`plot-renderer`, wrapped in `data-testid="ls-inline-plot"`) when `parsePlotFigure(plotJson)` succeeds, and falls back to the static PNG `InteractivePlot` otherwise — so a missing/broken `matslop_export_fig` doesn't break the UI.
- Plotly event subscription: after `Plotly.react(el, ...)` resolves, the div gains a jQuery-like `el.on(eventName, handler)` AND `el.emit(eventName, payload)` — the latter is the clean way to fire `plotly_click` / `plotly_doubleclick` / `plotly_clickannotation` from tests without synthesizing real pointer events. Pinned-annotation buckets are split: 2D goes in top-level `layout.annotations` (use `point.xaxis._name` / `point.yaxis._name` for xref/yref so subplots place correctly); 3D goes in `layout[sceneKey].annotations` with sceneKey from `point.data.scene` (defaults to `'scene'`). Set `captureevents: true` on an annotation so `plotly_clickannotation` fires when it's clicked.
- Figure export IPCs: `figures:saveDialog(defaultName)` returns `{filePath, format}` using the extension the user typed/picked; reuse it for any "save a plot" flow instead of rolling a new format chooser. `figures:exportPlot(filePath, data, encoding)` writes base64 PNG (stripping a `data:image/...;base64,` prefix) or UTF-8 SVG/text. Prefer `Plotly.toImage(el, {format, width, height})` over `Plotly.downloadImage` in Electron so the file goes through the IPC/dialog rather than the Chromium download manager. Preload bridge is writable at runtime — e2e tests stub `window.matslop.*` in `page.evaluate` to avoid native dialogs.
